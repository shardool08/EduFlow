# Pathshala AI — CLAUDE.md

## Project Overview
Pathshala AI is a voice-first, AI-powered English lesson planning companion for Grade 1 teachers in PCMC and NMC (Maharashtra) municipal corporation schools. It covers Marathi, Hindi, Urdu, and English medium schools.

**This is NOT a lesson plan generator.** It is a thinking partner that helps teachers co-plan lessons through voice conversation, building their analytical thinking for English curriculum over time.

## Tech Stack
- **Frontend:** React Native (Expo) — mobile-first, Android priority
- **Language:** TypeScript (strict mode)
- **State:** Zustand for local state, AsyncStorage for persistence
- **AI Backend:** Claude API (Sonnet) for lesson planning conversation
- **Speech-to-Text:** Google Cloud Speech-to-Text (Marathi, Hindi, English)
- **PDF Generation:** Server-side via Puppeteer + HTML templates (Node.js/Express backend)
- **Database:** Supabase (PostgreSQL + Auth + Storage)
- **Distribution:** WhatsApp share via native Share API

## Critical Design Rules

### Voice-First UX
- Microphone button is ALWAYS the largest, most prominent element on any input screen
- Every text input field has a mic button as the PRIMARY input method
- Typing is a fallback, never the expectation
- Voice input supports Marathi, Hindi, English — including code-mixed speech
- After voice transcription, ALWAYS show the transcribed text for teacher confirmation before proceeding

### Language & Bilingual UI
- All UI text lives in `i18n/` translation files — NEVER hardcode strings
- Support 4 languages: Marathi (mr), Hindi (hi), Urdu (ur), English (en)
- Urdu requires RTL layout support and Nastaliq script (use Noto Nastaliq Urdu font)
- AI conversation outputs are bilingual: teacher instructions in L1, English content for children
- Include explicit code-switching cues in lesson plans: "आता मुलांना मराठीत समजावून सांगा..."

### Designed for 40+ Age Group
- Minimum touch target: 56dp (larger than Android default 48dp)
- Minimum Devanagari font size: 18sp; Latin: 16sp
- NO hamburger menus, NO swipe gestures, NO hidden interactions
- ONE primary action per screen
- High contrast: no light gray text on white backgrounds
- Every screen must be navigable with: large visible buttons, clear back navigation, obvious primary action

### TLM Philosophy: Audit, Don't Generate
- NEVER generate printable TLMs (flashcards, picture cards)
- During planning, ASK the teacher what TLMs she has
- If critical TLMs are missing, GUIDE her to make them physically
- This is a core design principle, not a shortcut

### Lesson Wrapper Structure (40 minutes)
Every lesson plan follows this exact structure:
1. Hook aligned to objective (5 min)
2. Prior knowledge check (3-5 min)
3. Core teaching (5-10 min) — short because Grade 1
4. Practice — Guided/Individual/Group (10-15 min) — AI recommends mode
5. Check for understanding (3-5 min)
6. Exit tokens (2-3 min) — individual check per child
7. Closure (2-3 min)
Classroom management tips are woven throughout, NOT a separate section.

### Printable Outputs
- Lesson plan PDF: A4, bilingual, teacher's name in header, "Made with Pathshala AI" footer
- Worksheet: A4, child-facing, opt-out (generated by default, teacher can skip)
- Assessment sheet: A4, opt-in (teacher explicitly requests)
- All PDFs must be B&W-safe (photocopier-friendly)
- Fonts: Noto Sans Devanagari, Noto Nastaliq Urdu, Noto Sans (Latin)

## Code Standards
- Use functional components with hooks (no class components)
- Use TypeScript strict mode — no `any` types
- All user-facing strings go through i18n — check with `grep -r "\"[A-Za-z]" app/ components/` to find hardcoded strings
- Test every screen on a 5.5" Android viewport (360x640dp)
- Accessibility: all interactive elements need `accessibilityLabel` in the current language
- Error states: every async operation needs loading, success, and error states — never leave the user staring at a blank screen

## Verification
Before marking any feature complete:
1. Run `npx tsc --noEmit` — zero type errors
2. Run `npx eslint . --ext .ts,.tsx` — zero lint errors  
3. Run `npm test` — all tests pass
4. Manually test on Android emulator at 360x640dp viewport
5. Check that all strings are in i18n files (no hardcoded text)
6. Verify voice input works for that screen (if applicable)
7. Verify RTL layout for Urdu (if applicable)
